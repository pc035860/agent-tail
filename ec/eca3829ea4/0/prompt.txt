## Efficiency Review: `feat/nested-subagent-display`

### HIGH

**H1. `readSubagentMeta` worst-case latency = 300ms per registration** (`src/claude/subagent-detector.ts:123-146`)
Loop bound is `attempt &lt;= retries` (i.e. 6 iterations when `retries=5`), with a sleep on every non-final iteration. Worst case for a missing meta.json: 6 stat-checks + 5×50ms sleeps = **250ms**, not 250ms inclusive. Then `lookupParentWithRetry` adds up to another **150ms** sequentially (4 attempts × 50ms × 3 sleeps). Total worst case to register a subagent whose meta.json never lands: **~400ms** of timer-blocked latency before `tryAddSubagentFile` is even scheduled (which itself adds `retryConfig.initialDelay`).

Since `finalizeRegistration` is fire-and-forget per agent, parallel spawns each pay this independently — fine for concurrency, but the per-agent floor for `--pane` open is now noticeably higher than before. **Recommendation**: lower default `retries` to 3 (≈150ms cap) for the common-case main-spawn where meta.json should already exist at write-time, or skip meta entirely when `queuedDescription` is present AND label is being built without a parent (MAIN spawn path). Per your context: "always read meta.json" was the fix — the trade-off (added latency on the hot path) is not documented in the code, only in the inline comment about FIFO drift.

**H2. `scheduleSubagentsDirRetry` polls forever with no backoff** (`subagent-detector.ts:581-590`)
For a session that never spawns subagents (or where the user passed `--with-subagents` speculatively), this fires every 1000ms for the entire session lifetime. Each tick re-enters `tryWatchSubagentsDir` → catches → falls into `watchParentForSubagentsDir` → catches → reschedules. **Recommendation**: exponential backoff capped at e.g. 30s, or cap at N attempts (Claude Code typically creates the dir within the first few seconds if it's going to). Even a simple jitter+cap would prevent the indefinite 1Hz timer churn for sessions that never produce subagents.

### MEDIUM

**M1. `readSubagentMeta` uses TOCTOU exists+text** (`subagent-detector.ts:132-134`)
`Bun.file(metaPath).exists()` then `.text()` = two syscalls per attempt. Call `.text()` directly and catch `ENOENT` in the existing `try/catch`. Saves one syscall per attempt × up to 6 attempts × N subagents. Bonus: removes a real (if minor) TOCTOU window.

**M2. Sequential meta-read → parent-lookup is theoretically parallelizable, but as you noted depends on `meta.toolUseId`** — confirmed not parallelizable. Skip.

### LOW

**L1. `recordSpawn` Map grows unbounded for session lifetime** (`subagent-detector.ts:307, 483-485`)
~36 bytes/entry × total Task tool_uses. For a long-running session with hundreds of Task calls, this is still &lt;100KB. Cleared in `stop()`. Acceptable.

**L2. Hot-path per-line cost negligible** (`watch-builder.ts:166-174`) — `parsed.isTaskToolUse &amp;&amp; parsed.taskToolUseId` is two property reads; `extractAgentIdFromLabel` is a `slice` + `indexOf`. Per-line cost is dominated by JSON.parse upstream. Skip.

### MEDIUM

2. **`scheduleSubagentsDirRetry` (`:583-592`) vs `scheduleScan` (`:508-529`) vs `handleEarlyDetection` setTimeout (`:353-374`)** all use the same idiom: `isWatching` guard → `setTimeout` → `pendingTimers.add/delete`.
