### C2 — `SnapshotWatcher.reload()` lastJson 更新時機與 plan 文字矛盾

Plan Step 6 implementation note (b) 寫：
&gt; `JSON.parse(text)` in try/catch — JSON.parse failure calls `onError(err)`, does NOT call `onChange`
&gt; update `lastJson = text` only after successful parse (so next valid write retries)

但 SPEC §8.2 reference code（SPEC.md:688-697）是：
```typescript
const json = await readFile(this.config.path, 'utf8');

```
SPEC 是「先 update `lastJson`」，plan 寫「after successful parse」。兩者**行為不同**——關鍵是 plan 的 test case「Recovery after invalid JSON: Write valid JSON → onChange called」會驗證 plan 版本（正確的恢復語意），但 SPEC reference code 會**永久跳過該同樣 invalid text 的下一次相同寫入**（dedup 把無效輸入 cache 進去）。

## Warnings (should fix)

### W1 — Trivially-passing "rejects" tests pair `expect(...).toBe(false)` / `.toBeNull()` with stubs that return `false` / `null`
- `paths.test.ts:115,119,123,127,128` — every `isValidWorkflowRunId('...')).toBe(false)` line currently passes only because the stub returns `false`. The discriminator is **only** the single positive case at line 111.
- `paths.test.ts:138,142,146,150` — same shape for `isValidWorkflowAgentId`.
- `paths.test.ts:162,166,170,183,188,193` — every `parseWorkflow*Filename(...)).toBeNull()` passes trivially against stubs.
- `journal-parser.test.ts:82–129` — six `parse(...)).toBeNull()` tolerance tests pass trivially.
- **Why it matters**: When a GREEN candidate implementation lands, these tests do not differentiate between "correctly rejects bad input" and "incorrectly rejects everything". A naive impl that always returns `null` from `parseWorkflowAgentFilename` would pass 6/7 tests in that block.
- **Rationale**: The positive cases (`paths.test.ts:111,134,156,176`; `journal-parser.test.ts:21–30`) still anchor the regex shape, so the situation isn't catastrophic, but the asymmetry is real (positive : negative = ~1:4 for several helpers).
- **Recommendation**: For each pair, also assert one **boundary positive** that the stub would still fail (e.g., for `isValidWorkflowRunId`, add `wf_00000000-000` and `wf_ffffffff-fff` to lock both ends of the hex range; for `parseWorkflowSnapshotFilename`, assert two distinct runIds parse correctly so the positive isn't a single hardcoded constant). This buys mutation resistance for free.

### W2 — `cwdToClaudeProjectFilter('')` returns `''` — both stub and expected impl agree
- `paths.test.ts:90–92`: `expect(cwdToClaudeProjectFilter('')).toBe('')` passes against the stub.
- **Why it matters**: Same trivially-passing class as W1. The 4 other positive cases (lines 84–106) still anchor behavior, so this is the weakest case of W1, not a blocker.

### W3 — `journal-parser` truncation assertion is loose
- `journal-parser.test.ts:56–61`: asserts `formatted.length &lt; longBody.length + 100` and `formatted.includes('…')`. With the planned 100-char cap, the formatted string is roughly `'✓ agent xxxxxxx result: ' + xxx + '…'` ≈ 30+100+1 ≈ 131 chars. The assertion `&lt; 200 + 100 = 300` is satisfied by **any** output that contains `'…'` and is &lt; 300 chars — even an implementation that truncates at 250 chars would pass.
- **Recommendation**: Tighten to `expect(parsed!.formatted.length).toBeLessThan(200)` (or `&lt; 150`). That actually exercises the SPEC §8.1 "truncate to ~100 chars" contract. Also verify the truncated content matches the spec: `expect(parsed!.formatted).toMatch(/x{100}…/)` or similar.

### W4 — `journal-parser` "result body number" assertion under-specifies SPEC §8.1
- `journal-parser.test.ts:73–78`: asserts `parsed!.formatted.includes('42')`. SPEC §8.1 (and plan Step 5 line 176) says the formatted result is `'✓ agent {short} result: ' + safeStringify(...)`. The test would pass if a buggy impl printed `42` anywhere — e.g., in a stray `agentId` or a debug field. Same for the `'null'` test (line 64–70).
- **Recommendation**: Pair with `expect(parsed!.formatted).toContain('result: 42')` (or `'result: null'`) to anchor the formatting contract, not just substring presence.

### W5 — `snapshot-watcher` cross-platform `fs.watch` reliability not addressed in tests
- `snapshot-watcher.test.ts` relies on `fs.watch(path, ...)` firing on subsequent `writeFile()` overwrites. Per CLAUDE.md §19.6 and SPEC's own gotcha note, `fs.watch` is unreliable on macOS for atomic rename, and `writeFile` on some platforms uses temp+rename semantics. Tests use direct `writeFile` (overwrite), which is the documented good path — but the test gives no slack for the **flake risk** plan §R2 calls out.
- **Recommendation**: Either (a) document the flake-tuning lever (`DEBOUNCE_MS * 3 + 50`) inline as the plan suggests, or (b) increase `DEBOUNCE_MS` in CI mode. Not a correctness issue; a stability issue for the upcoming GREEN run.

### W6 — `snapshot-watcher.test.ts:223–247` "onChange throwing does not crash" couples to a specific implementation decision
- The test asserts `errors.length &gt;= 1` from a **synchronous** throw inside `onChange`. Per the plan's Step 6 point 2, `onChange` is wrapped in try/catch in `reload()` and forwarded to `onError`. That's a SPEC-aligned contract.
- However, the *initial* `start()` path also runs `reload()` synchronously, so the test implicitly assumes the `start()` itself surfaces the throw (line 235 `await watcher.start();` then asserts errors at 238). This is a brittle ordering assumption — if a future impl moves the initial read into a microtask, the assertion `errCountBefore = errors.length` at line 240 could already be 0 when the second write fires.
- **Recommendation**: Replace `errCountBefore = errors.length` with `await waitDebounce()` between the two writes; then assert `errors.length &gt;= 2`. Decouples from "did the initial read finish surfacing errors before line 240?".

### W7 — `types.compile.test.ts` runtime assertions are dead weight
- e.g. `expect(_p.workflowEvent).toBe('started')` at line 36 — this is just confirming what was literally typed two lines above. The value comes from the test source itself, not from the type system.
- **Why it matters**: Mild test smell — the comment at line 14–16 ("Test bodies do nothing at runtime; the value is in TypeScript type-checking") is then contradicted by the runtime assertions. Either remove the `expect`s entirely (TS compile failure is the only signal needed) or drop the misleading comment.
- **Rationale**: Low impact (the compile-time gate is the actual contract) but invites future devs to add runtime assertions and miss the point.

## Risks &amp; mitigations

- **Risk**: `fs.watch` on `transcriptDir` fires before `agent-*.jsonl` is fully written → `attachAgent` reads empty file. **Mitigation**: `FileWatcher` retries on empty content via its internal polling (existing). But the plan's `attachAgent` doesn't retry on `watcher.start()` failure. The rollback path is there, but on failure no auto-retry happens — relying on the next fs.watch event. For P3 this is acceptable (matches Cursor's `tryAddSubagentFile` rollback pattern), but document this in the plan: "rollback only — retry depends on next fs.watch event."

- **Risk**: Commander's `--no-X` semantics surprise. **Mitigation**: Patch 3a forces matching keys.

- **Risk**: macOS `fs.watch` may not fire for files in a transcriptDir created milliseconds after dir watch starts (race). **Mitigation**: SPEC §19.6 acknowledges this is a known issue. P3 plan correctly does `_scanAndAttachInitialAgents` BEFORE `_startSubagentDirWatch`, which catches files already on disk before the watch races in.

- **Risk**: P4 SnapshotWatcher integration might want to insert BEFORE step 3 (dir watch) if snapshot completion needs to suppress new agent attaches. **Mitigation**: SPEC §10.2 R3-B4 is explicit — snapshot is Step 4. P3 plan leaves correct slot. No action needed in P3; just document the wedge point.

- **Risk**: `process.once('SIGINT')` in `startClaudeWorkflowMultiWatch` swallows SIGINT for the rest of the process. Other code paths register `SIGINT` handlers too (check `src/index.ts:1437+`). **Mitigation**: Use `process.on('SIGINT', ...)` and add `await attachment.stop` to the existing shutdown flow if one exists, or scope the handler with cleanup.
