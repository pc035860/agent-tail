## Critical issues (must fix)

### C1 — `WorkflowDetectorConfig` 在 plan 漏掉 SPEC §6.2 的 `outputHandler: OutputHandler` 欄位

Plan Step 3 的 `WorkflowDetectorConfig` 寫成：
```typescript
export interface WorkflowDetectorConfig {
  sessionUuid: string;
  sessionsRoot: string;
  onNewWorkflow: (workflow: DetectedWorkflow) =&gt; void;
  // OutputHandler import lives in src/core/detector-interfaces.ts; P3 will wire.
}
```
但 SPEC §6.2（SPEC.md:343-350）明定：
```typescript
export interface WorkflowDetectorConfig {
  sessionUuid: string;
  sessionsRoot: string;
  onNewWorkflow: (workflow: DetectedWorkflow) =&gt; void;
  outputHandler: OutputHandler;   // ← MISSING in plan
}
```
而且 §9.1 path B、§10.2 都呼叫 `this.config.outputHandler.debug(...)` / `.info(...)`，P3 一動工就會編譯錯。Plan 註解說「P3 will wire」是錯的——SPEC 已敲定該欄位，不該被 punt 到 P3。

**Fix**: 直接在 P1 加入 `outputHandler: OutputHandler` 欄位並 import `OutputHandler` from `../core/detector-interfaces.ts`。這正是 plan 自己宣稱 D8「Forward-declare for stable shape」的初衷。

### C2 — `SnapshotWatcher.reload()` lastJson 更新時機與 plan 文字矛盾

Plan Step 6 implementation note (b) 寫：
&gt; `JSON.parse(text)` in try/catch — JSON.parse failure calls `onError(err)`, does NOT call `onChange`
&gt; update `lastJson = text` only after successful parse (so next valid write retries)

但 SPEC §8.2 reference code（SPEC.md:688-697）是：
```typescript
const json = await readFile(this.config.path, 'utf8');
if (json === this.lastJson) return;
this.lastJson = json;                           // ← BEFORE JSON.parse
const snapshot = JSON.parse(json) as ...;
this.config.onChange(snapshot);
```
SPEC 是「先 update `lastJson`」，plan 寫「after successful parse」。兩者**行為不同**——關鍵是 plan 的 test case「Recovery after invalid JSON: Write valid JSON → onChange called」會驗證 plan 版本（正確的恢復語意），但 SPEC reference code 會**永久跳過該同樣 invalid text 的下一次相同寫入**（dedup 把無效輸入 cache 進去）。

Plan 版本對使用者更友善（invalid → 不污染 dedup cache → 下次有效寫入會處理）。**建議 plan 採用「after successful parse」並在 D6 補一句註：偏離 SPEC §8.2 reference code，理由是無效 text 不應污染 dedup**。否則 P1 與 SPEC text matching review 會被質問。

### C3 — `JournalLineParser` constructor 的 `verbose?: boolean` 沒有實際被使用，與 SPEC §8.1 不符

Plan Step 5 寫 `new JournalLineParser({ fileMtime?: Date, verbose?: boolean } = {})`，但 SPEC §8.1（SPEC.md:609）只列 `constructor(opts: { fileMtime?: Date } = {})`——**沒有 verbose**。而且 plan 的 implementation 描述沒任何地方真的用到 `verbose`（truncation 是硬寫 100 字元 cap，不是 verbose-conditional）。

**Fix**: 拿掉 constructor 的 `verbose?: boolean` 參數，**或者**保留但於 plan 中明確說明 truncation 應該在 `verbose === true` 時跳過（與 `src/utils/text.ts:truncate({ verbose })` 介面一致）。後者更實用、與既有 `ClaudeLineParser` 慣例對齊，但要更新 implementation 描述與 test cases。

### C4 — Plan 提到的 `truncateLine` helper 在 codebase 不存在

Plan Step 5 implementation 提到 `truncateLine(event.result, 100)`，但 `src/utils/text.ts` 只有 `truncate` 與 `truncateByLines`（兩者簽名都不是 `(str, n)`）。Plan 後面 D5 又說「inline 3-line `truncate(str, n)`」。**前後不一致**。

**Fix**: 統一寫法。建議 P1 加一個 module-local helper：
```typescript
function truncateLine(str: string, max: number): string {
  return str.length &lt;= max ? str : str.slice(0, max) + '…';
}
```
並在 plan 中明確：「不重用 `src/utils/text.ts:truncate`（它走 head+tail+omitted-msg 路徑，與 journal 100-char cap 語意不同）」。

### C5 — `parseWorkflowSnapshotFilename` 在 paths.ts 沒對應的 spec 來源，但 plan 的 ID 正則寬鬆

Plan Step 2 的 `WORKFLOW_RUN_ID_REGEX = /^wf_[0-9a-f]{8}-[0-9a-f]{3}$/` 正確（與 SPEC §3.1、§7.2、§9.1.1 的 `wf_{8hex}-{3hex}` 一致）。**但 plan 沒驗證 SPEC §3.4 的 17-hex agentId 是 lowercase**——SPEC.md:83-84 與 §3.4 都隱含 lowercase hex（Claude Code 實際輸出皆為小寫），plan 的 `/^[0-9a-f]{17}$/` 正確，**但測試裡缺少「reject uppercase 17-hex」case**（reject uppercase 只在 `isValidWorkflowRunId` 出現一次）。

**Fix**: 在 `tests/claude-workflow/paths.test.ts` 補一行 `isValidWorkflowAgentId('ADCA0C33EBE734C2D')` → `false` 的 case。順手在 `parseWorkflow*Filename` 的測試也補 uppercase 字串 → null。

## Test plan

Plan 提供的 test list 覆蓋面已經很好（paths 19 cases + journal-parser 12 cases + snapshot-watcher 8 cases），建議**增補**：

### `paths.test.ts` 補強
- `isValidWorkflowAgentId('ADCA0C33EBE734C2D')` → `false`（uppercase 17-hex）
- `cwdToClaudeProjectFilter('/')` → `'-'`
- `cwdToClaudeProjectFilter('Users/x')` → `'Users-x'`（無前綴 `/`，相對路徑）
- `getWorkflowAgentPath` 用 invalid agentId（短於 17 hex）— plan 沒約定要 throw 或 silently 接受。**建議 plan 明確**：路徑助手不做 ID 驗證（純字串拼接），caller 自負；測試以「接受任意字串只做拼接」驗證。

### `journal-parser.test.ts` 補強
- `result` 為 `null` → formatted 含 `null` 或 `''`（不 crash）
- `result` 為 cyclic object → 不 crash（觸發 try/catch path，見 R6）
- `result` 為 number `42` → `JSON.stringify(42) === '42'`，formatted 含 `: 42`
- History mode 連續 parse 5 行 → 5 個 ParsedLine 的 `timestamp` 完全相同（驗證 historyTimestamp 不被誤清）
- `markLiveMode()` 後 parse → timestamp ≠ history value（plan 已有，但加一個「parse 連續兩行間隔 5ms → 兩個 timestamp 不同」驗證 live mode 拿真實 wall clock）

### `snapshot-watcher.test.ts` 補強
- **`onChange` callback 拋例外** → SnapshotWatcher 不 crash 整個 process（用 try/catch 包 onChange 呼叫，或 plan 明確說「onChange must not throw — caller responsibility」）。
- **`start()` 在路徑不存在時** → `onError` 被呼叫一次，不 throw。Plan 沒提路徑不存在情境。
- **`onError` callback 缺省（undefined）** → invalid JSON 仍不 crash（plan 寫 `onError?:`，要有 case 驗證）。

### 新增 `types.compile.test.ts`（plan 列為 optional）
**建議升級為 required**——成本極低，但能在 P3+ 的 refactor 中早期捕捉 ParsedLine / SessionListItem optional field 被誤改 required 的 regression。
